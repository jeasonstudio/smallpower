# Organizing meeting reports (整理会议报告)

## 概述

在整理会议内容时，默认执行者对当前项目与业务背景几乎零上下文。将用户提供的原始材料（会议转录、会议文档、会议纪要、`*.srt`）整理成可直接分发的语义化文档：会议章节大纲、发言人观点总结、决策结论、行动项、风险与待确认问题。

除非执行者明确提到会议内容与当前工作空间或代码仓库无关，否则你需要默认将会议内容与当前工作空间的代码上下文进行关联。文档需要足够完整，让未参会的开发成员也能快速理解“发生了什么、决定了什么、接下来谁做什么”。输出要分块清晰、语义清晰、可快速扫描，避免冗余（DRY），避免无关扩展（YAGNI），只保留对执行有价值的信息。

默认假设执行者写作能力和工程能力良好，但对当前项目术语、人员分工和上下文不熟悉。因此必须补足必要背景，不得依赖“大家都懂”的隐含信息。

* **开始时声明**：`我正在使用 organizing-meeting-reports 技能整理会议内容并生成报告。`
* **上下文**：建议在独立工作区中执行整理任务，避免与并行开发任务互相干扰。
* **保存路径**：`docs/meeting-reports/YYYYMMDD-<会议主题>/<会议报告文档>.md`

## 用户输入

在继续之前您**必须考虑**用户输入（如果不为空）。

## 何时使用

- 用户提出“整理项目文档”等基于会议内容的文档整理需求。
- 用户提出会议总结、会议大纲或其他会议相关结构化输出需求。
- 用户提供 `*.srt` 等会议记录文件并需要内容分析与归纳。
- 用户提供会议转录文本并需要提取关键信息。
- 用户提供会议纪要或会议文档并需要重新组织成更清晰的报告。






## 工作流

1. 提取会议目标、关键决策、行动项、风险与未解决问题。
2. 合并重复信息，并统一人名、模块名、日期等表达。
3. 将模糊表述转为可执行任务。
4. 基于章节内容提炼“发言人核心观点”，按角色聚合其主张、关注点与结论倾向。
5. 按下方模板输出整理结果；需要“会议章节（大纲）”时优先复用模板引用中的章节模板。

## 模板引用

- 会议章节（大纲）：
  - 模板：`templates/meeting-chapter-outline.md`
  - 用途：将会议内容按时间段拆分成章节，快速产出结构化大纲。
  - 字段规范：每章至少包含 `时间`、`标题`、`参与人`、`概要`。
  - 拆分建议：按讨论自然转折拆分，建议 3-10 个章节，避免过细或过粗。
  - 内容要求：`标题`概括该时段核心结论；`概要`建议 2-4 句，覆盖背景、分歧、结论与影响。
  - 观点要求：模板末尾必须包含 `发言人核心观点梳理`，按发言人汇总核心观点条目。
  - 说明：模板文件本身是完整演示示例；实际产出时请复用其结构，不要直接复制示例内容。

## 报告模板

```markdown
# 会议报告 - YYYY-MM-DD

## 摘要

- ...

## 决策结论

- [D1] ...

## 行动项

- [ ] 负责人: @name | 任务: ... | 截止日期: YYYY-MM-DD

## 风险 / 阻塞

- ...

## 待确认问题

- ...
```

## 质量标准

- 每个行动项都必须包含负责人和截止日期（未知可写 `TBD`）。
- 报告应简洁、易于快速浏览。
- 缺失信息必须记录到“待确认问题”。
- 发言人核心观点需与章节内容一致，不得凭空补充会议中未出现的结论。
- 每位核心发言人至少提炼 2 条观点，且观点表述应可追溯到对应章节。
